Security

▶ Critical No authentication or authorization enforced on connections server.rs:serve_connection

The per-connection handler has a TODO comment acknowledging that peer fingerprint authorization is not implemented. Any client with a valid TLS certificate can connect and access any share with full read access. The README describes a permission system (.allow files per share with fingerprint + access level), but none of it is enforced at runtime.

// TODO(v1): authorise peer fingerprint against per-share permission files // before accepting any streams.

Additionally, the RiftWelcome hardcodes read_only: true regardless of config, so the protocol itself can't communicate the correct access level even if auth was added.

Implement peer fingerprint validation in serve_connection before accepting any streams. Load per-share permission files on startup. Reject connections that don't match any allow rule. Derive read_only from the permission file for the matched fingerprint.

▶ Critical Memory exhaustion via unbounded file reads handler.rs:read_response

read_response calls tokio::fs::read(&canonical) which reads the entire file into memory before slicing it into chunks. A malicious or misbehaving client can request a read on a multi-gigabyte file, causing the server process to allocate that much RAM, potentially triggering OOM. There is no file size limit, no streaming, and no backpressure.

let content = match tokio::fs::read(&canonical).await { Ok(c) => c, // Entire file loaded into heap … };

Stream file data using tokio::fs::File + tokio::io::BufReader. Read chunks on demand based on byte offsets rather than loading the whole file. Add a configurable max file size limit (e.g., 4 GiB). This also eliminates the duplicate Merkle tree computation in read_response vs. get_or_compute_merkle_root.

▶ Critical TOCTOU race in path canonicalization — symlink insertion attack handler.rs:resolve, lookup_response

The resolve() function looks up a path from the HandleDatabase, then canonicalizes it to check it stays within the share root. This is correct at the moment of the check, but creates a Time-of-Check-to-Time-of-Use (TOCTOU) race: between canonicalization and the actual file operation (stat, read, etc.), an attacker with local filesystem access could replace a legitimate file or directory with a symlink pointing outside the share.

// Step 1: Canonicalize resolves symlinks, verifies inside share let canonical = tokio::fs::canonicalize(&stored_path).await?; // --- RACE WINDOW: attacker replaces file with symlink here --- // Step 2: Open the file using the now-stale canonical path let meta = tokio::fs::metadata(&canonical).await?; // may follow new symlink

Open the file first via File::open, then call file.metadata() on the file handle (not the path) to get metadata for the opened file. Use std::fs::File and then file.metadata() / file.canonicalize() on Linux via /proc/self/fd/.... Alternatively, use O_NOFOLLOW on open to refuse symlinks at the leaf.

▶ High No rate limiting or concurrency limits — DoS vector server.rs:accept_loop, serve_connection

Both accept_loop and serve_connection spawn a new tokio::spawn task for every incoming connection and every stream without any bounds. A malicious client can open thousands of connections or streams, each of which triggers filesystem I/O and Merkle tree computation. This can exhaust server resources (memory, CPU, file descriptors) very quickly.

Add a Semaphore or tokio::sync::Semaphore around task spawning with a configurable max concurrency (e.g., 256 connections, 1024 streams total). Consider per-client-fingerprint rate limiting. Back-pressure the accept loop when limits are reached.

▶ High xattr handle forgery — unauthenticated persistent handles handle.rs:get_or_create_handle

The HandleDatabase persists UUID handles as extended attributes (user.rift.handle) on files. Any process with write access to the filesystem can set this xattr to a chosen UUID, effectively injecting arbitrary handle mappings that the server will trust on next lookup. The server reads the xattr and trusts it without verification.

let handle = match xattr::get(path, RIFT_HANDLE_XATTR) { Ok(Some(value)) if value.len() == 16 => { Uuid::from_slice(&value).unwrap_or_else(|_| Uuid::now_v7()) } … };

If a user sets user.rift.handle to collide with the UUID of a different file's handle, the second file's get_or_create_handle will find a pre-existing entry in the map and return the wrong file's UUID. The insert-then-re-lookup on Exists pattern does prevent one collision path, but only for concurrent inserts — not for xattr-forged collisions.

Sign or HMAC the xattr value with a server key before writing, and verify the signature on read. Alternatively, don't read xattrs at all and generate handles purely in-memory, rebuilding from disk only via populate_from_share at startup.

▶ High Self-signed cert auto-generation with hardcoded CN and no key rotation cert.rs:generate_and_save_cert

The cert auto-generation uses rcgen::generate_simple_self_signed(vec!["rift-server"]) which produces a cert with a hardcoded CN of "rift-server", no Subject Alternative Names, and a default validity period. There is no mechanism to rotate keys — once generated, the same cert/key is reused forever. A leaked private key compromises all past and future connections with no remediation path.

Add SANs for the server's hostname/IP. Make the CN configurable. Add a --rotate-cert flag or automatic rotation based on expiry. Consider adding a key rotation protocol where the server can publish a new cert fingerprint to authorized clients.

▶ Medium No input validation on start_chunk / chunk_count in READ_REQUEST handler.rs:read_response

The read_response handler accepts start_chunk and chunk_count from the client without validation. A client can send start_chunk = u32::MAX or chunk_count = u32::MAX, which will cause .skip() and .take() on the chunk list. While this doesn't panic, a very large chunk_count combined with a file that has many chunks could cause a long iteration and many frame sends.

Add an upper bound on chunk_count (e.g., 256). Clamp or reject out-of-range start_chunk values early before any file I/O.

▶ Medium Silent error swallowing hides security events handler.rs (multiple), merkle.rs (multiple)

Multiple locations silently discard errors with let _ = .... Most notably:

  • let _ = database.put_merkle(...).await; — DB write failures are silently ignored
  • let _ = xattr::set(path, RIFT_HANDLE_XATTR, handle.as_bytes()); — xattr write failures ignored
  • let _ = database.put_tree(...).await; — tree persistence failures ignored

While this prevents crashes, it means that cache corruption, disk-full errors, or permission problems are invisible to operators. This is especially concerning for security-sensitive operations where cache integrity matters (Merkle tree tampering detection).

Log at warn! level when cache writes fail. Consider surfacing persistent write failures as metrics. For xattr writes on directories, the silent skip is arguably fine, but on files it should at least be logged.

Design & Architecture

▶ High Only the first share is exported — multi-share config is misleading main.rs:63

The config and CLI support defining multiple shares, but main.rs only passes server_config.shares[0].path to the server. The remaining shares are silently ignored, and there's no warning that they're dropped. The handshake also only sends one ShareInfo.

let share_path = server_config.shares[0].path.clone(); // Only first share!

Either implement multi-share support (pass all shares to accept_loop, dispatch by share name in the handshake), or validate and reject configs with more than one share with a clear error message. The current silent truncation is the worst of both worlds.

▶ High Arc<Option<Database>> — awkward type erasure main.rs, server.rs, handler.rs

The database is threaded through as Arc<Option<Database>>, requiring unwrapping at every call site with db.as_ref().as_ref(). This conflates two concerns: shared ownership (Arc) and optionality (Option). The current pattern forces every handler to handle the "no database" case, creating code duplication.

let db_ref = db.as_ref().as_ref(); // Double unwrap — appears in every handler

Replace with a trait object or Arc<dyn MerkleCache> where MerkleCache is implemented by both Database and a NoopCache / NullCache type. This eliminates Option from the type entirely and removes the double-unwrap from every handler call site.

▶ Medium No filesystem abstraction — direct tokio::fs calls in handlers handler.rs (throughout)

All handlers call tokio::fs::* directly. This couples business logic to the physical filesystem and makes it impossible to test with virtual/mock filesystems. The current tests rely on tempfile::TempDir, which means they touch the real disk and are slow/non-deterministic. Any future support for S3-backed shares, FUSE mounts, or encrypted-at-rest storage would require major refactoring.

Introduce a Vfs trait with methods like stat, read_dir, open, read, etc. Implement it for the real filesystem. Inject the trait into handlers. This also makes it possible to add per-share VFS implementations (e.g., read-only enforcement at the VFS layer).

▶ Medium Inconsistent handler error handling — some return types, some stream directly handler.rs:stat_response vs read_response

Handlers have two different error handling patterns:

  • Return type pattern: stat_response, lookup_response, readdir_response — return a proto struct with an error variant. Clean and testable.
  • Stream pattern: read_response, merkle_drill_response — take &mut S: RiftStream and write error frames directly. Harder to test, harder to reason about.

Unify on one pattern. The return-type pattern is better for testability and composability. For streaming handlers, separate the "resolve and prepare" phase (returns a result) from the "send frames" phase (takes a stream). This makes the streaming handlers testable without a stream.

▶ Medium No connection state — handshake results are not tracked server.rs:serve_connection

The handshake produces a RiftWelcome with important state (protocol version, share info, root handle), but this state is not stored anywhere. Subsequent streams on the same connection don't know which share was selected, what protocol version was negotiated, or what capabilities were agreed upon. This makes it impossible to enforce per-connection constraints (e.g., a v1 client shouldn't send v2 messages).

Create a ConnectionState struct that captures the handshake result. Store it wrapped in Arc and pass it to stream handlers. This also enables future per-connection features like negotiated capabilities, session timeouts, and authorization scope.

▶ Low No graceful shutdown / drain logic main.rs:tokio::select!

When ctrl-c is pressed, the server immediately drops the listener and all in-flight tasks. There's no drain logic to wait for active streams to complete, which means in-flight READ operations are silently aborted mid-transfer.

Use a CancellationToken. On ctrl-c, stop accepting new connections/streams but wait for existing streams to complete with a timeout (e.g., 30 seconds). Log in-flight stream count during shutdown.

▶ Low Unknown message types silently close stream with no error to client server.rs:handle_stream
other => { // TODO(v1): send ERROR_RESPONSE so clients surface a real error. debug!("unknown message type 0x{:02X} — closing stream", other); }

Clients that send an unsupported message type get the stream silently closed with no error frame. This makes debugging client-side issues very hard. The TODO has been present since the initial implementation.

Define a generic ERROR_RESPONSE message in the protocol. Send it before closing the stream for unknown types. This is a 5-line fix.

Code Quality & Maintainability

▶ High handler.rs is a 1,169-line monolith with 5 distinct handlers + helpers handler.rs

handler.rs contains: resolve(), metadata_to_attrs(), build_attrs(), stat_response(), lookup_response(), readdir_response(), read_response(), merkle_drill_response(), get_or_compute_merkle_root(), and numerous helper functions. At 1,169 lines it's well beyond the typical ~300-line guideline for a single Rust file. Each handler has its own error helpers and they share no common structure.

Split into: handler/mod.rs (dispatch + resolve), handler/stat.rs, handler/lookup.rs, handler/readdir.rs, handler/read.rs, handler/merkle.rs. Move metadata_to_attrs and build_attrs into an attrs.rs. Add a shared error.rs for common error helpers.

▶ High Duplicate Merkle tree computation in read_response vs stat handler.rs:read_response, get_or_compute_merkle_root

read_response computes the full Merkle tree to determine chunk boundaries and hashes, then computes it again at the end to store in the cache. get_or_compute_merkle_root does the same work independently. If a client first does STAT then READ on the same file, the Merkle tree is computed twice for the same content.

Extract a compute_merkle_tree() function that returns both the root hash and the full tree structure. Use it in both STAT and READ paths. Consider caching the result in a small LRU cache keyed by (path, mtime) since the same file may be accessed multiple times in quick succession.

▶ Medium Schema SQL duplicated between open() and open_in_memory() metadata/db.rs

The three CREATE TABLE statements are copy-pasted between Database::open() and Database::open_in_memory(). If either is updated without the other, the schemas will silently drift. This is a DRY violation and a maintenance hazard.

// In open(): conn.execute_batch("CREATE TABLE IF NOT EXISTS merkle_cache (...); CREATE TABLE ..."); // In open_in_memory(): ← exact same string, copy-pasted conn.execute_batch("CREATE TABLE IF NOT EXISTS merkle_cache (...); CREATE TABLE ...");

Extract the schema DDL into a const SCHEMA: &str or a fn init_schema(conn) helper that both constructors call.

▶ Medium Inconsistent error codes for similar failure modes handler.rs

Handlers use different error codes for logically similar conditions:

  • Malformed payload: stat_response returns empty results, lookup returns ErrorUnsupported, readdir returns ErrorUnsupported
  • Invalid UUID handle: stat returns ErrorNotFound, lookup returns ErrorNotFound, read returns ErrorNotFound — this is consistent ✓
  • Path escapes share: resolve bails with a string error, which maps to ErrorNotFound sometimes and ErrorPermissionDenied never — despite being a security violation

Add a ErrorCode::ErrorPermissionDenied for path traversal attempts (which is a security event, not "not found"). Use consistent error codes for malformed payloads (e.g., ErrorCode::ErrorMalformed across all handlers). Log path traversal attempts at warn! level.

▶ Low Magic number: max_concurrent_streams: 128 server.rs:handle_stream (RIFT_HELLO handler)

The RiftWelcome message hardcodes max_concurrent_streams: 128 with no way to configure it and no enforcement of the limit on either side.

Make this configurable via ServerConfig. Enforce the limit in serve_connection by tracking active streams with a Semaphore.

▶ Low Outdated walkdir = "1" dependency Cargo.toml

The crate depends on walkdir = "1" which was released in 2015. The current version is 2.x with API improvements and bug fixes. Version 1 is effectively unmaintained.

Upgrade to walkdir = "2". The API surface is similar but has minor breaking changes (e.g., WalkDir::new returns a different iterator type). This is a low-effort change.

Correctness

▶ High put_tree writes empty leaf_hashes to legacy cache — data inconsistency metadata/merkle.rs:put_tree

When put_tree updates the legacy merkle_cache table for backward compatibility, it writes empty bytes ("".as_bytes()) for the leaf_hashes column. If get_merkle later hits this entry (which it will, since the mtime/size match), it will try to deserialize empty bytes as leaf hashes, silently fail, and return None. This means the cache entry is created but can never be read back successfully — it's a write-only cache entry that wastes space and forces re-computation.

// In put_tree(): tx.execute( "INSERT OR REPLACE INTO merkle_cache ... VALUES (?1, ?2, ?3, ?4, ?5, ?6)", rusqlite::params![path_str, mtime, file_size, root_bytes, "".as_bytes(), now], );

Either serialize the real leaf_hashes into the legacy cache entry, or skip the legacy cache insert entirely in put_tree (the tree nodes table is authoritative). If keeping it for backward compat, at least set a flag so get_merkle knows to check the tree nodes table.

▶ Medium lookup_response re-canonicalizes the share root on every call handler.rs:lookup_response

lookup_response calls tokio::fs::canonicalize(share) and tokio::fs::canonicalize(&child_path) on every request. The share root canonicalization is redundant — it's the same value every time and adds an unnecessary syscall. More importantly, if the share root becomes unreachable (e.g., NFS down), every lookup fails even for files within an already-canonicalized subtree.

Canonicalize the share root once at startup and store it in the ConnectionState struct. Pass it into handlers instead of the raw share path.

▶ Medium Merkle cache keyed by file path string — rename breaks cache metadata/db.rs, merkle.rs

The SQLite schema uses file_path TEXT PRIMARY KEY across all three tables. If a file is renamed, the old cache entry becomes orphaned (never matched) and a new one is created on next access. The old entry is never cleaned up. Over time, this leads to unbounded cache growth. Additionally, if a different file is later created at the same path with the same mtime/size, it could receive the wrong cached Merkle tree.

Consider adding an inode/device-number key to the cache to make entries truly identity-based. Add a periodic cleanup job or LRU eviction to bound cache size. The mtime/size staleness check mitigates the "wrong cache" risk but doesn't eliminate it for adversarial scenarios.

▶ Low HandleDatabase xattr read uses non-canonical path handle.rs:get_or_create_handle

get_or_create_handle first canonicalizes the path and stores the canonical path in the map. But then it reads/writes xattrs using the original path argument, not the canonical one. If the input path and the canonical path differ (e.g., due to symlinks in intermediate components), the xattr is read from the symlink path but the in-memory map uses the resolved path. This inconsistency could cause handles to be lost or re-created unnecessarily on server restart.

let canonical = tokio::fs::canonicalize(path).await?; // … map uses `canonical` … let handle = match xattr::get(path, RIFT_HANDLE_XATTR) { // ← non-canonical! … }; // and: if path.is_file() { let _ = xattr::set(path, RIFT_HANDLE_XATTR, handle.as_bytes()); // ← non-canonical! }

Use canonical for both xattr reads and writes. Add a canonical.as_os_str() call instead of the raw path argument.

ℹ️

Observations

▶ Info No database schema versioning or migration system metadata/db.rs

The schema is created with CREATE TABLE IF NOT EXISTS and has no version tracking. Adding columns or tables in future versions would silently create them alongside the old schema, but there's no way to migrate existing data or detect schema version mismatches. If a column is renamed or a constraint changes, the old table persists unchanged.

Add a schema_version INTEGER table (or pragma) and check it on open. Implement a simple migration runner. This is low priority now but will become critical as the schema evolves.

▶ Info Extensive but partially duplicated test suite (2,430 lines) tests/server.rs

The integration test file is very thorough (2430 lines!) with good coverage of security cases (path traversal, malformed payloads, symlink escapes). However, there's significant overlap with the unit tests in handler.rs tests module. Many test cases test the exact same scenarios at different levels, which increases maintenance burden without proportional value.

Consolidate: keep unit tests for edge cases and error paths in handler.rs. Keep integration tests only for protocol-level concerns (framing, handshake, multi-stream). Add a brief comment at the top of each test module explaining what level it tests and why.

▶ Info Pre-epoch mtime handling was fixed, but other edge cases remain metadata/merkle.rs:get_merkle

The codebase correctly handles pre-epoch mtime (the test at metadata/merkle.rs:get_merkle_handles_pre_epoch_mtime proves this was a past bug). Good. However, there are other edge cases untested: files with mtime at exactly UNIX_EPOCH (0), files where meta.modified() returns Err (e.g., some filesystems don't support mtime), and files where as_nanos() as u64 overflows for very large mtime values.

Consider adding a mtime_ns == 0 guard to always re-compute (since 0 could mean "unsupported" or "actual epoch"). This avoids caching entries that might be wrong.

▶ Info Merkle drill rebuilds the full tree on every request (no lazy loading) handler.rs:merkle_drill_response

merkle_drill_response reads the entire file, builds the full Merkle tree with cache, and stores it to the database — on every drill request. This is correct for the first drill on a file, but wasteful for subsequent drills where the tree is already in the DB. The code does check the DB for children, but only after rebuilding the tree.

Check the cache first: if get_merkle returns a hit, the tree should already be in the DB from a previous put_tree call. Skip the tree rebuild in that case and go straight to get_children.

✅

What's Done Well

  • Path traversal protection via canonicalization + share-prefix check in resolve() is correct at the logical level and handles both direct and symlink escapes.
  • Handle eviction on stale/escaped paths — resolve() removes invalid handles from the HandleDatabase, preventing stale entries from accumulating.
  • Robust malformed payload handling — every handler catches decode() errors and returns an error response, never panicking on bad input.
  • Concurrent insert safety — the get_or_create_handle insert-then-re-lookup pattern correctly handles races, verified by the 100-concurrent-call test.
  • Good test coverage for security cases — symlink escapes, path traversal, malformed payloads, and pre-epoch mtime are all tested.
  • SQLite WAL mode — correct choice for concurrent read-heavy workload.
  • Clean separation between transport (server.rs) and business logic (handler.rs), even if handler.rs is large.
