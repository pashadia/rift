NEW 17 findings
S3 TOCTOU race in path canonicalization — symlink insertion attack handler.rs:resolve
Between canonicalize() and the subsequent file operation, an attacker with local filesystem access could replace a file with a symlink pointing outside the share. The canonical path is validated but then used on an already-resolved path that may have changed. No bead exists for this. This is different from the resolved rift-9l0 (drill bypassing resolve entirely) — this is a race within resolve itself.
S4 No rate limiting or concurrency limits — unbounded task spawning server.rs:accept_loop, serve_connection
Both connection and stream handlers use bare tokio::spawn with no Semaphore or bound. A malicious client can exhaust memory, FDs, and CPU. No bead covers DoS protection or concurrency limits for the accept/stream loops.
S5 xattr handle forgery — unauthenticated persistent handles handle.rs:get_or_create_handle
Any process with filesystem write access can set user.rift.handle to a chosen UUID, injecting handle mappings the server will trust. rift-3ak covers silent xattr write failures — this is a different (and more severe) issue: unauthenticated xattr reads that enable handle injection and potential UUID collision attacks.
S7 No input validation on start_chunk / chunk_count in READ_REQUEST handler.rs:read_response
start_chunk and chunk_count are taken from the client without bounds checking. A client can request chunk_count = u32::MAX, triggering an extremely long iteration and many frame sends. No bead covers input validation on read request parameters.
D2 Arc<Option<Database>> — awkward double-unwrap type erasure main.rs, server.rs, handler.rs
db.as_ref().as_ref() appears in every handler. The Option conflate shared ownership with optionality. Should be replaced with a trait object (Arc<dyn MerkleCache>) implemented by both Database and a NoopCache. No bead exists for this refactoring.
D4 Inconsistent handler error patterns — return types vs. stream writes handler.rs
stat/lookup/readdir return proto structs (testable, clean). read/merkle_drill take &mut S: RiftStream and write frames directly (hard to test, mixes concerns). No bead for unifying handler error patterns.
D5 No connection state — handshake results not tracked server.rs:serve_connection
The handshake produces a RiftWelcome with protocol version, share info, root handle — but none of this is stored. Subsequent streams don't know which share was selected or what version was negotiated. No bead for per-connection state tracking.
D6 No graceful shutdown / drain logic main.rs:tokio::select!
Ctrl-C drops the listener and all in-flight tasks immediately. No CancellationToken, no drain timeout, no in-flight stream tracking. No bead for graceful shutdown.
D7 Unknown message types close stream with no error to client server.rs:handle_stream
The code has a TODO(v1): send ERROR_RESPONSE comment but no bead was filed for implementing it. Clients get a silent stream close on unknown message types, making debugging very hard.
Q1 handler.rs is a 1,169-line monolith with 5 handlers + helpers handler.rs
rift-772.6 covers moving inline tests to tests/server.rs, but no bead exists for the module splitting effort itself (stat.rs, lookup.rs, readdir.rs, read.rs, merkle.rs, attrs.rs, error.rs).
Q3 Schema SQL duplicated between open() and open_in_memory() metadata/db.rs
Three CREATE TABLE statements are copy-pasted between the two constructors. Schema drift risk. No bead for deduplicating DB schema DDL.
Q4 Inconsistent error codes for similar failure modes handler.rs
Malformed payloads: stat → empty results, lookup → ErrorUnsupported, readdir → ErrorUnsupported. Path traversal: always ErrorNotFound (should be ErrorPermissionDenied). No bead for error code consistency.
Q5 Magic number: max_concurrent_streams: 128 server.rs:handle_stream
Hardcoded in RiftWelcome, not configurable, not enforced. No bead for making this configurable or enforcing the limit.
Q6 Outdated walkdir = "1" dependency Cargo.toml
walkdir v1 is from 2015 and unmaintained. Current is v2. No bead for dependency upgrades.
C1 put_tree writes empty leaf_hashes to legacy cache — data inconsistency metadata/merkle.rs:put_tree
put_tree inserts "".as_bytes() for leaf_hashes into the legacy merkle_cache table. When get_merkle later hits this entry, deserialization silently fails and returns None — the cache entry can never be read back. rift-ms7 is about build_with_cache() producing length: 0 in rift-common; this is a different bug in rift-server's put_tree writing garbage to the legacy column.
C2 lookup_response re-canonicalizes share root on every call handler.rs:lookup_response
Unnecessary tokio::fs::canonicalize(share) syscall on every lookup. Should be computed once at startup. No bead for this redundant I/O.
C4 HandleDatabase xattr read uses non-canonical path handle.rs:get_or_create_handle
The map stores canonical paths, but xattr reads/writes use the original path argument. If input and canonical paths differ (symlinks), handles may be lost on restart. rift-3ak is about silent xattr failures — this is a separate issue (wrong path used for xattr ops).

S1 No authentication or authorization enforced on connections server.rs:serve_connection
Related beads:  rift-3lh rift-5dt rift-9k6

The beads describe the feature work to implement auth (fingerprint validation, per-share permissions, TLS client certs). What the beads don't convey is the security criticality: right now any client with any TLS cert gets full read access to all shares. The TODO comment in serve_connection and the hardcoded read_only: true in RiftWelcome are specific code-level gaps not mentioned in any bead. Additionally, the auth beads are P1 but unblocked by unresolved predecessor chains — the urgency of the current state is not captured.
S6 Self-signed cert with hardcoded CN, no SAN, no key rotation cert.rs:generate_and_save_cert
Related beads:  rift-5j9 (PEM support) rift-7rk (auto-renewal)

Neither bead covers the security implications of the current implementation. rift-5j9 is about format support (PEM vs DER). rift-7rk is about renewal before expiry. Neither addresses: (1) hardcoded CN "rift-server" with no SAN, (2) no key rotation mechanism — a leaked private key compromises all connections with no remediation, (3) no way to configure the cert's subject identity. These are distinct security gaps beyond format and renewal.
S8 Silent error swallowing hides security events handler.rs, merkle.rs (multiple)
Related bead:  rift-3ak (xattr write failures)

Report broadens scope beyond xattrs. rift-3ak covers silent xattr write failures only. My report identifies 3 additional categories of silently swallowed errors: (1) let _ = database.put_merkle(...).await — cache write failures in handler.rs, (2) let _ = database.put_tree(...).await — tree persistence failures in handler.rs, (3) all DB write failures in merkle.rs. Cache integrity failures are especially concerning for Merkle tree tampering detection.
Q2 Duplicate Merkle tree computation in stat vs. read paths handler.rs:read_response, get_or_compute_merkle_root
Related bead:  rift-qn6 (full file read + tree rebuild on every MerkleDrill)

Report identifies additional duplication paths. rift-qn6 focuses on the merkle_drill_response path. My report identifies a second duplication: stat_response → get_or_compute_merkle_root computes the tree, then read_response on the same file computes it again independently. A typical client workflow (stat then read) doubles the work. Additionally, read_response computes the tree twice within the same function (once for chunk boundaries, once for cache storage at the end).
D3 No filesystem abstraction — direct tokio::fs calls handler.rs
Related bead:  rift-4i2 (pluggable storage backends)

Bead is P4 and much broader — misses the immediate testability/coupling concern. rift-4i2 is about abstracting storage behind a trait for S3/object-store backends (P4, far-future). My report frames the issue at a different level: immediate concerns about testability (can't use mock filesystems), per-share VFS enforcement (read-only at VFS layer), and the coupling that makes any future storage change a major refactor. A lightweight Vfs trait is a nearer-term need than the full pluggable backend system.
Category 3

Reported in Beads but Lacking Context — Report Fills the Gap

2 findings
D1 Only the first share is exported — multi-share config is misleading main.rs:63
Bead:  rift-uvi — "Server currently only serves first share from config. Need to wire all shares."

What the bead misses: The bead is P3 and describes it as a missing feature. My report identifies that this is actually a high-severity issue because:
  • The config parser accepts multiple shares silently — no warning, no error
  • Users who configure 3 shares will only get 1, with no indication the others were dropped
  • The handshake sends only one ShareInfo, so clients can't even discover other shares
  • shares[0] indexing will panic if config has zero shares (though main.rs has a guard for empty shares before this line)
The bead should be upgraded to P2 at minimum, or at least reject multi-share configs with a clear error message until multi-share support is implemented.
C3 / I1 Merkle cache keyed by file path string; no schema versioning metadata/db.rs, merkle.rs
Related bead:  rift-xay (configurable cache size limits with LRU eviction)

What the bead misses: rift-xay focuses on eviction policy (LRU, size limits). My report identifies a separate correctness issue with the cache key design:
  • file_path TEXT PRIMARY KEY — renamed files orphan their cache entries; a new file at the same path with matching mtime/size could get the wrong Merkle tree
  • No schema_version table — no way to migrate existing data or detect version mismatches on open
  • Orphaned entries are never cleaned up → unbounded growth even without LRU
These are data integrity concerns, not just sizing — LRU eviction alone won't prevent the wrong-cache-hit scenario.

Relevant Beads Not Surfaced by This Review

3 beads in rift-server scope
— rift-0ao — Fragile string matching for QueryReturnedNoRows
Multiple methods in merkle.rs check for "Query returned no rows" by string-matching the error message. My review noticed the silent error handling but didn't call out this specific fragile pattern-matching issue. The bead is correct — e.to_string().contains("Query returned no rows") will break if tokio_rusqlite changes its error message format. Worth creating a bead-level fix.
— rift-wqx — Streaming directory pagination for readdir
My review noted that readdir_response collects all entries into a Vec before returning, but didn't flag this as a separate memory concern for directories with 100k+ entries. The bead correctly identifies this as a streaming/pagination issue. My review's focus was on the handler-level design; the pagination concern is additive.
— rift-ms7 — Single-leaf build_with_cache() produces length: 0
This bug is in rift-common's MerkleTree::build_with_cache(), not rift-server directly. My review scope was rift-server only, so it wasn't surfaced. However, it's related to my finding C1 (put_tree empty leaf_hashes) in that both produce incorrect length metadata in the cache layer. They're distinct bugs but in the same subsystem.
