### 1. rift-0ao — Fix fragile string matching for QueryReturnedNoRows

**Priority:** P2 | **Effort:** ~1 hour | **File:** `metadata/merkle.rs`

**Problem:** Three methods in `merkle.rs` detect "no rows" by string-matching `e.to_string().contains("Query returned no rows")`. This breaks if `tokio_rusqlite` or `rusqlite` changes its error message formatting.

**What to do:**
1. Check whether `tokio_rusqlite::Error` wraps `rusqlite::Error` in a way that allows pattern matching on the inner variant (e.g., `rusqlite::Error::QueryReturnedNoRows`). It likely does — `tokio_rusqlite::Error` wraps `rusqlite` errors.
2. If yes: replace all three string-match sites with proper variant matching:
   ```rust
   Err(e) => match e {
       tokio_rusqlite::Error::Rusqlite(rusqlite::Error::QueryReturnedNoRows) => Ok(None),
       other => Err(other),
   }
   ```
3. If tokio_rusqlite doesn't expose the inner variant cleanly: create a small helper `fn is_no_rows(e: &tokio_rusqlite::Error) -> bool` that does the check in one place, and call it from all three sites. Add a test that verifies the detection works against both the string and the error type.
4. Affected methods: `get_merkle()`, `get_children()`, `get_leaf_info()` in `merkle.rs`

**Acceptance:** No `to_string().contains("Query returned no rows")` remains in the codebase. Existing tests pass.

### 2. rift-5j9 — Add PEM certificate support to the server

**Priority:** P2 | **Effort:** ~2 hours | **File:** `cert.rs`

**Problem:** `cert.rs:read_cert_and_key()` detects PEM but rejects it with a "PEM not supported" error. Users who generate certs with `openssl` (which outputs PEM by default) can't use them without manual DER conversion.

**What to do:**
1. Add `rustls-pemfile` (or `pem` crate) to `Cargo.toml` dependencies
2. In `read_cert_and_key()`, before the DER validation path, detect PEM format and parse it:
   ```rust
   if is_pem_encoded(&cert) {
       let cert_der = pem_to_der(&cert)?;
       let key_der = pem_to_der(&key)?;
       return Ok((cert_der, key_der));
   }
   ```
3. Implement `pem_to_der()` using `rustls_pemfile::certs()` / `rustls_pemfile::private_key()` — strip PEM armor, decode base64, return DER bytes.
4. Update `use_cert_and_key()` — it currently bails on PEM. Remove the PEM bail since we now handle it above.
5. Add tests: create a PEM cert with `rcgen`, write it in PEM format, verify `get_or_create_cert` reads it correctly.
6. Update the error message in `read_cert_and_key()` — the current "rift-server only supports DER format" text should no longer say that.

**Acceptance:** Server reads PEM certs/keys generated by `openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem -days 365 -nodes`. Existing DER tests still pass.

### 3. rift-vhp — Add input validation on read request chunk params

**Priority:** P1 | **Effort:** ~30 min | **File:** `handler.rs:read_response()` (lines ~575-600)

**Problem:** `start_chunk` and `chunk_count` are used directly from the client request without bounds checking. A client can request `chunk_count = u32::MAX`, causing extremely long iteration and massive frame generation.

**What to do:**
1. After decoding `ReadRequest`, add validation before any file I/O:
   ```rust
   const MAX_CHUNK_COUNT: u32 = 256;
   if req.chunk_count > MAX_CHUNK_COUNT {
       // send ErrorInvalidRequest response and return
   }
   ```
2. If `start_chunk > chunk_boundaries.len()`, send an error response with `ErrorNotFound` (the range is out of bounds — no chunks to read) rather than silently returning a 0-chunk success. This makes client debugging easier.
3. Add a test: send `ReadRequest { start_chunk: 0, chunk_count: u32::MAX }` and verify the server returns an error (not OOM or a 4-billion-frame response).
4. Add a test: send `start_chunk` past the end of the file's chunks and verify a proper error.

**Acceptance:** `chunk_count > 256` returns an error. `start_chunk` beyond file's chunk count returns an error. Existing read tests pass. Add the `MAX_CHUNK_COUNT` const as `pub` so it can be referenced from the client side.

### 4. rift-ev4j — Use canonical path for xattr operations in HandleDatabase

**Priority:** P2 | **Effort:** ~30 min | **File:** `handle.rs:get_or_create_handle()` (lines 85-100)

**Problem:** `xattr::get()` and `xattr::set()` are called on the raw `path` argument, but the in-memory map stores the canonical path. When these differ (symlinks in intermediate components), handles may not persist across restarts.

**What to do:**
1. In `get_or_create_handle()`, after computing `canonical`, use `canonical` instead of `path` for xattr operations:
   ```rust
   // BEFORE:
   let handle = match xattr::get(path, RIFT_HANDLE_XATTR) { ... };
   let _ = xattr::set(path, RIFT_HANDLE_XATTR, handle.as_bytes());

   // AFTER:
   let handle = match xattr::get(&canonical, RIFT_HANDLE_XATTR) { ... };
   let _ = xattr::set(&canonical, RIFT_HANDLE_XATTR, handle.as_bytes());
   ```
2. Also change the `path.is_file()` check to `canonical.is_file()` (since canonical is the resolved path).
3. Add a test: create a symlink to a directory, call `get_or_create_handle` via the symlink path, verify the xattr is written to the canonical (target) file, not the symlink entry.

**Acceptance:** `xattr::get` and `xattr::set` are called with the canonical path. The `path.is_file()` check uses canonical. Test using symlink path verifies xattr lands on the target.

### 5. rift-8y1m — Deduplicate schema SQL in Database

**Priority:** P2 | **Effort:** ~15 min | **File:** `metadata/db.rs`

**Problem:** The three `CREATE TABLE` statements are copy-pasted between `Database::open()` and `Database::open_in_memory()`. Schema drift risk.

**What to do:**
1. Extract the DDL into a constant:
   ```rust
   const SCHEMA_DDL: &str = "
       PRAGMA journal_mode=WAL;
       PRAGMA foreign_keys=ON;
       CREATE TABLE IF NOT EXISTS merkle_cache (...);
       CREATE TABLE IF NOT EXISTS merkle_tree_nodes (...);
       CREATE TABLE IF NOT EXISTS merkle_leaf_info (...);
   ";
   ```
   (Note: WAL + foreign_keys are shared between open() and open_in_memory(), so include them in the DDL string too. For `open_in_memory()`, WAL is a no-op which is fine.)
2. Add a helper:
   ```rust
   fn init_schema(conn: &mut rusqlite::Connection) -> rusqlite::Result<()> {
       conn.execute_batch(SCHEMA_DDL)
   }
   ```
3. Replace both `conn.call(|conn| { conn.execute_batch(...) })` blocks with `conn.call(|conn| { init_schema(conn); Ok(()) })`.
4. Verify existing tests pass — they exercise both constructors.

**Acceptance:** `CREATE TABLE` appears exactly once in the file. `open()` and `open_in_memory()` both call the same init function. All existing tests pass.

### 6. rift-bp2 — Send ERROR_RESPONSE for unknown message types

**Priority:** P2 | **Effort:** ~15 min | **File:** `server.rs:handle_stream()`

**Problem:** Unknown message types cause a silent stream close with no error to the client. The `TODO(v1)` has been sitting there since initial implementation.

**What to do:**
1. In the `other =>` match arm in `handle_stream()`, instead of just logging and dropping, send an error response before closing:
   ```rust
   other => {
       debug!("unknown message type 0x{other:02X} — closing stream");
       let error = ErrorDetail {
           code: ErrorCode::ErrorUnsupported as i32,
           message: format!("unknown message type 0x{other:02X}"),
           metadata: None,
       };
       // Stream may be in any state; best-effort send, ignore errors
       let _ = stream.send_frame(msg::ERROR_RESPONSE, &error.encode_to_vec()).await;
       let _ = stream.finish_send().await;
   }
   ```
2. Verify `msg::ERROR_RESPONSE` exists in `rift-protocol` (if not, define it as the next available type ID).
3. Add a test: open a stream, send a frame with an unused type ID, verify you receive an ERROR_RESPONSE frame with `ErrorUnsupported` before the stream closes. (The existing `server_rejects_wrong_protocol_version` test in `tests/server.rs` is a similar pattern — extend it or add a new one.)

**Acceptance:** Unknown message type → server sends ERROR_RESPONSE with ErrorUnsupported code → stream closes. Test verifies the error frame is received.
